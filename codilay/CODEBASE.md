## Overview
CodiLay is a sophisticated suite of tools designed for processing codebases and managing related documentation through both AI-driven insights and user-friendly interfaces. This project includes a command-line interface (CLI), a configuration system, several data processing modules, and a documentation management system. The primary architectural components are distributed across various Python modules within the `src/codilay` directory. The CLI system, anchored by `codilay_legacy.py` and `src/codilay/__main__.py`, enables user interaction by leveraging Python's `click` and `rich` libraries for command management and enhanced terminal output. The project manages state and configuration persistence utilizing JSON-based settings and dataclass patterns.

Key Components:
- **Processor System**: Automates code processing into documentation pieces by working with chunks of code, handled in `src/codilay/processor.py`.
- **Wire Management**: Tracks dependencies and connections between components, focused in `src/codilay/wire_manager.py`.
- **LLM Client Integration**: Interfaces with language models to enable advanced natural language processing (LLM) tasks, seen in `src/codilay/llm_client.py`.
- **Interactive Menu System**: Richly designed terminal interactions are managed via the `rich` library, implemented in `src/codilay/menu.py` and `src/codilay/ui.py`.

Key entry points for new developers include understanding the configuration settings in `codilay.config.json` and engaging with the CLI to perform tasks like documentation generation and code triage. The project also leverages external libraries such as `openai`, `anthropic`, and `tiktoken` for robust NLP operations.

## .gitattributes Configuration
> File: `.gitattributes`

The `.gitattributes` file is used to manage Git's behavior with regards to line ending normalization and file handling within the repository. It provides rules for handling text files and binary files.

### Text File Normalization
- The command `* text=auto` enables automatic detection of text files and standardizes line endings to LF (Line Feed).
- Specific file types such as `*.py`, `*.md`, `*.json`, `*.toml`, `*.txt`, and `*.sh` have explicit rules applied to ensure LF line endings, which is essential for consistency across different development environments.

### Binary Files
- Files with extensions like `*.png`, `*.jpg`, `*.jpeg`, `*.gif`, `*.ico`, `*.mov`, `*.mp4`, and `*.pwf` are marked as binary. This prevents Git from attempting to normalize line endings for these files, which could lead to file corruption. By denoting them as binary, the integrity of these files is maintained.

This configuration is crucial for maintaining cross-platform consistency and ensuring the correct handling of files by Git.

## Introduction to CodiLay
> File: `README.md`

### 🦅 CodiLay

CodiLay is an AI-driven documentation tool designed to dynamically generate, update, and maintain comprehensive documentation for codebases. It is particularly focused on tracing 'wires,' which are unresolved references in code, to provide a persistent and searchable knowledge base. This is achieved through **The Wire Model**, which treats every import, function call, and variable reference as a Wire to be resolved.

CodiLay provides both a Command Line Interface (CLI) for interacting with your documentation process and a Web User Interface (UI) for a more graphical approach. Its features include interactive chat support, git-aware incremental updates, and the ability to run complex triage to classify files by their documentation needs.

By deploying a multi-stage approach that includes smart triage, chat-based interaction, and web-based browsing, CodiLay presents a holistic and intelligent solution to documentation management, aimed at enhancing both developer productivity and codebase comprehension.

## LLM Client Integration
> File: `src/codilay/llm_client.py`

The `LLMClient` class in `src/codilay/llm_client.py` provides a unified interface to interact with several language model providers, including Anthropic and OpenAI, plus OpenAI-compatible services. This allows the CodiLay system to work seamlessly with multiple AI backends depending on configuration.

#### Provider Registration

Providers and their configurations are stored in the `PROVIDER_CONFIGS` dictionary. Each entry includes details such as the SDK to use (`anthropic` or `openai`), environment variables for API keys, default models, and base URLs.

#### Initialization

The client is initialized with a configuration object that specifies the LLM provider among other settings. Upon initialization, the correct SDK client is created based on the provider type (`anthropic` or `openai`), using environment settings and overrides from the configuration.

#### Rate Limiting

To deal with rate-limit errors, the client implements retry logic. It identifies specific rate-limit error classes from the `anthropic` and `openai` packages, and retries calls with increasing wait times.

#### API Calls

The `call` method serves as the main interface for sending prompts to the model. It provides automatic handling for JSON responses and retries on network or parsing errors. Specific methods handle raw API calls to Anthropic or OpenAI, routing appropriately based on the provider.

#### Token Counting and Usage Stats

The client tracks token usage and can provide statistics via `get_usage_stats`. Tokenization is handled by the `tiktoken` package, attempting to use model-specific encodings, with fallback options if unavailable.



### Configuration Dependency
The `LLMClient` relies on configurations provided by `CodiLayConfig`, such as `llm_model`, `llm_provider`, and `llm_base_url` for setting up the language model interactions.



### Multi-Provider Support
This feature enhances the `LLMClient` to accommodate multiple providers, leveraging OpenAI-compatible APIs and SDK routing. It ensures seamless integration across various environments by managing API key requirements and model specifications dynamically.

### Interaction with ChatStore

The `ChatStore` utilizes the LLMClient for generating reformatted documentation sections and extracting memory through conversational analysis.

## File Chunking System
> File: `src/codilay/chunker.py`

### `Chunker` Class
This class is responsible for splitting large files into manageable chunks for processing. It implements different strategies based on file type, such as splitting on class or function definitions for Python, JavaScript/TypeScript, Dart, Java/Kotlin, and Go files. This file-specific approach enhances the relevance and usability of the chunks. 

#### Construction
The `Chunker` class is initialized with a `token_counter` callable for counting tokens within file content and a `config` object with threshold settings for chunking.

- `count_tokens`: Counts tokens in a provided text block to help determine chunk sizes.
- `threshold`: The token limit at which a file will be considered for chunking.
- `max_chunk_tokens`: The maximum number of tokens allowed per chunk.
- `overlap_ratio`: Determines the overlap between consecutive chunks to preserve context.

#### Plan Method
The `plan` method generates a `ChunkPlan` for a given file, which may result in a single `FULL` chunk if the file is small enough, or a `SKELETON` chunk and multiple `DETAIL` chunks if chunking is necessary.

- Uses a threshold to decide if a file requires chunking, ensuring efficient processing.
- Skeleton extraction identifies imports and signatures without bodies for an overview.
- Structural boundaries are detected based on the file extension to identify logical divisions.
- Overlap is added between chunks to ensure contextual continuity.
- Token counts for chunks are recalculated to fit within predefined limits.

#### Supporting Methods
Key methods include:
- `_extract_skeleton`: Gleans the structural essence from a file, focusing on imports, signatures, and documentation comments.
- `_find_boundaries`: Detects structural boundaries specific to different programming languages, which is crucial for content chunk algorithm.
- `_split_on_boundaries` and `_split_by_tokens`: Split content into chunks based on detected boundaries or by token count as a fallback.

These mechanisms ensure the file is split optimally based on its content and structure, preserving code contexts in a manageable form.

→ See `FileChunk` and `ChunkPlan` for more details on chunk representations.

#### Chunk Management Methods

These methods handle the creation and management of file chunks:

- **Chunk Breaking by Tokens:**
  The code segments the file's content into chunks based on a token threshold as defined by `max_chunk_tokens`. It aims to break at a broader scope like a blank line to maintain logical coherence within chunks.
  - **Carry Over Lines:** Any remaining lines after a chunk is formed are carried over to the next chunk for continued processing.

- **Appending Final Chunk:**
  The final set of lines is added as a chunk if not already processed, ensuring no part of the file is left unchunked.

- **`_add_overlap`:**
  This method is key in applying overlaps between chunks. By taking a portion of the previous chunk and prepending it to the current one, continuity is preserved across the document when processing out of context segments.
  - **Overlapping Strategy:** Overlaps are calculated as 10% of the prior chunk's line count and capped at 30 lines.

- **Chunk Subdivision - `_sub_split_chunk`:**
  This utility deals with oversized chunks by dividing them further while relabeling parts to maintain order. Useful for managing large sections that exceed token limits within a single chunk.
  
#### Helper Methods
Technical helper methods are provided for various language syntax recognitions, such as identifying import lines, signatures, comments, decorators, and export statements. These heuristics enable language-agnostic processing by detecting:
  - Language-specific constructs like `class`, `def`, `import`, etc.
  - Comment and documentation markers across different file types, ensuring only relevant portions are chunked.

## Retriever System
> File: `src/codilay/retriever.py`

### Retriever System
The `Retriever` class in `src/codilay/retriever.py` is designed for efficient section retrieval within a chat context, specifically targeting the inclusion of relevant parts from a documentation corpus while minimizing token usage.

#### Functionality
- **Purpose**: The Retriever prioritizes chat context relevance using TF-IDF scoring, narrowing down the context to the 3-5 most pertinent sections. This economizes token usage by 75-85% per query.

#### Key Components
- **`ScoredSection` Dataclass**: Provides a structure for document sections with associated relevance scores and a method for formatting sections for language model context injection.
- **Tokenizer**: A private function `_tokenize` to split text into lowercase tokens, filtering out common stop words. It supports camelCase and snake_case splitting, essential for diverse naming conventions in codebases.
- **TF-IDF Index**: The `Retriever` leverages an internal structure that builds a TF-IDF index across documentation sections, calculating document frequency and inverse document frequency for meaningful scoring.

#### Methods
- **`search`**: Scores and sorts sections using TF-IDF cosine similarity, delivering the top-k pertinent results.
- **`search_by_file` and `search_by_tags`**: Methods tailored to retrieve document sections by specific files or tags, enhancing targeted searches.
- **`build_context`**: Constructs a contextual response within a specified token budget, prioritizing full content inclusion but intelligently truncating when necessary.
- **`get_source_files`**: Retrieves file paths highly relevant to a query, facilitating deep agent file selection.

#### Internal Operations
- **Cosine Scoring**: Uses TF-IDF weighted vectors to calculate similarity scores between a query and document sections, utilizing both query and document term frequencies.

This component is critical for optimizing contextual data ingested by a language model during chat interactions, ensuring efficiency and relevance.

The `Scanner` class is responsible for scanning a codebase directory, identifying files that should be processed, and respecting various ignore patterns including `.gitignore` and configuration-specific rules.

### Key Features

- **Ignore Rules**: Constructs an ignore specification by reading patterns from a `.gitignore` file and additional custom ignore patterns specified in the configuration. The `pathspec` library is used to manage these patterns efficiently.

- **Text File Identification**: Differentiates text files from binary files by checking file extensions, specific filenames, and analyzing file contents for non-text indicators like null bytes.

- **File Scanning**: Implements `get_all_files` to recursively walk through the target directory, applying ignore rules and collecting files that should be processed based on size and text status.

- **File Tree Representation**: Offers `get_file_tree`, which attempts to generate a visual representation of the directory using the `tree` command or a custom tree-building method if the command fails.

- **Markdown Preloading**: Supports preloading markdown files with `preload_md_files`, allowing these files to be accessed and processed efficiently, crucial for documentation purposes.

- **Change Detection**: Incorporates the `get_changed_files` method to detect changes in files using Git, contributing to the system's ability to keep documentation up-to-date with code changes.

- **File Hashing**: Provides functionality to compute hashes of file contents, aiding in the detection of file modifications when other methods are unavailable.

### Architectural Decisions

Using `pathspec` for path pattern matching ensures flexibility and compatibility with multiple ignore file formats, supporting a wide array of programming languages and configuration styles within the codebase.

## Git Integration and Change Tracking
> File: `src/codilay/git_tracker.py`

The `git_tracker.py` module provides crucial integration with Git to track changes between documented states and the latest codebase. It facilitates continuous documentation by enabling precise change detection between specific commits and the current HEAD, as well as identifying uncommitted changes.

#### Components and Their Functions

- **`ChangeType` Enum**: Represents different types of file changes in a Git repository, such as `ADDED`, `MODIFIED`, `DELETED`, `RENAMED`, and `COPIED`.

- **`FileChange` Dataclass**: Describes an individual file change detected by Git, storing information such as the change type and file paths involved.

- **`GitDiffResult` Dataclass**: Aggregates a detailed analysis of changes between two commit states, providing properties to access added, modified, deleted, and renamed files.

- **`GitTracker` Class**: The core class responsible for interacting with the Git command-line tool to determine repository status, fetch commit information, and parse diffs. Key functions include:
  - `_run_git`: Executes Git commands via subprocess.
  - `get_diff`: Computes the changes between a base commit and the current state.
  - `get_uncommitted_changes`: Lists changes in the working tree yet to be committed.
  - `get_full_diff`: Compiles a comprehensive list of changes from a specified commit to the present working system, combining committed and uncommitted changes.

This module's ability to provide detailed summaries and track the state of files aligns with CodiLay's goal of maintaining up-to-date documentation.

#### Key Architectural Points

- Utilizes subprocess to execute git commands safely and efficiently.
- Provides methods to integrate seamlessly with the rest of the CodiLay system, especially through detailed tracking of file changes.
- Handles both committed and uncommitted changes, offering a holistic view relevant to documentation efforts.

## Triage System
> File: `src/codilay/triage.py`

### Triage System
The **Triage System** in CodiLay is responsible for the initial classification of files before any documentation processing occurs. It operates in two modes: **fast** and **smart**.

- **Fast mode** utilizes static patterns to instantly classify files without involving any Language Model (LLM). This mode is designed to be quick and cost-free.
- **Smart mode** leverages the LLM to analyze file structures and decide which files are significant for documentation, without sharing actual file content. This mode ensures more precise file categorization and is designed to be both efficient and cost-effective.

#### Components

- **`FileCategory`:** A class defining classifications like `core`, `skim`, and `skip` for organizing files based on processing priority.
- **`TriageResult`:** A dataclass capturing the output of the triage process. It includes lists of files in each category, the type of project detected, and the reasoning behind the classification.
  - **Methods**: include `move_to_core`, `move_to_skim`, and `move_to_skip` to adjust file lists dynamically based on classification needs.

#### Static Patterns
Static patterns are used both in fast mode and as a safety net in smart mode. These include directories and file extensions that should always be skipped and certain files to skim for a quick overview.

- **Skip Directories:** Includes `.git`, `node_modules`, and others not meant to be processed.
- **Skip Extensions:** Such as `.min.js`, `.map`, to ignore compiled or minified files.
- **Skim Files:** Important files like `package.json` are skimmed for metadata.

#### Triage Process
The `Triage` class initializes with optional `llm_client` and `config` parameters. The process involves:
- **`fast_triage`:** Uses static patterns for classification.
- **`smart_triage`:** Sends file tree data to an LLM to decide the classification based on the project context.

#### Utilities
- **Project Type Detection:** Uses common file indicators to classify project types, such as `flutter`, `django`, etc.
- **Token Savings Estimate:** Calculates tokens saved by skipping unnecessary files, optimizing the resource usage for the LLM.

This system enables efficient and scalable file classification, ensuring that only relevant parts of a codebase are processed, which is crucial for large projects or those with extensive external dependencies.

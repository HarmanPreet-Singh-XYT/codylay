## Overview
# CodiLay Project Overview

CodiLay is a sophisticated code documentation and management system designed to help developers and teams keep track of documentation across various stages of development. It provides a way to integrate language models (LLMs) into the software documentation process, offering capabilities such as automatic document generation, dependency management, and team collaboration enhancements.

## Architecture

The project is structured into several key modules and components, each serving distinct purposes:

- **Core Modules**:
  - **CodiLay CLI and Entry Points**: The primary entry point for CodiLay is the CLI found in `src/codilay/__main__.py`. It integrates with the `cli` module (`src/codilay/cli.py`) to facilitate command-line interactions.
  - **Pluggable Modules**: The architecture supports modular inclusion of components like `llm_client` for integrating various LLM APIs, `doc_differ` for change-tracking of documentation, and `wire_manager` to handle dependencies and references.
  - **Infrastructure Support**: CodiLay uses `git_tracker` to manage version control interactions and `scheduler` for automated tasks management.

- **Documentation Management**:
  - **DocStore**: The `src/codilay/docstore.py` module manages the storage and retrieval of documentation content, ensuring synchronization with the codebase and Git history.
  - **Exporter and Retrievers**: Components like `exporter` and `retriever` focus on document data handling, facilitating efficient access and export of valuable insights.

- **AI and Planning**:
  - **LLM Integration**: Through `llm_client`, CodiLay interfaces with language models to facilitate contextual understanding and documentation suggestions.
  - **Planning and Orchestration**: Involves the `planner` and `parallel_orchestrator` to ensure structured processing of documentation tasks.

- **User Interface**:
  - **CLI and UI**: The CLI extends a comprehensive interface for user interaction with CodiLay's capabilities. The `ui.py` module leverages rich console outputs for enhanced user experience.

- **Server and Extensions**:
  - **Hosting Server**: `server.py` provides API access, allowing external systems to interact with and benefit from CodiLay's capabilities, especially in chat and memory retrieval.
  - **VSCode Extension**: Supports integration with VSCode via an extension found under `vscode-extension`, enhancing IDE capabilities with CodiLay's functionalities.

- **Tests**:
  - Comprehensive testing is provided for various modules, ensuring reliable operation and facilitating continuous integration.

## For New Developers
New developers should begin by familiarizing themselves with the primary entry point (`src/codilay/__main__.py`) and resolve any setup dependencies defined in `setup.py`. From there, exploring key modules by following the tests and utilizing the VSCode extension can provide practical familiarity with CodiLay's capabilities and development process.

## Tech Stack
CodiLay is predominantly built with Python, leveraging technologies like FastAPI for server capabilities, and several LLM integration libraries. Dependency management is facilitated through `pyproject.toml` and `setup.py` for packaging.

## .gitattributes Configuration
> File: `.gitattributes`

The `.gitattributes` file is used to manage how Git treats certain file types within the repository. It plays a crucial role in ensuring consistent text file format across different environments and preventing binary files from being altered in a way that could corrupt them.

### File Handling Configuration

- **Automatic Line Ending Conversion**: 
  - `* text=auto`: Automatically detects text files and normalizes line endings to LF (Line Feed) to ensure consistency.

- **Specific File Type Configurations**: 
  - Files with the extensions `.py`, `.md`, `.json`, `.toml`, `.txt`, and `.sh` are explicitly set to use LF for line endings: `text eol=lf`.
  - Binary files such as `.png`, `.jpg`, `.jpeg`, `.gif`, `.ico`, `.mov`, `.mp4`, and `.pwf` are marked as `binary` to prevent line-ending conversion, which could corrupt these files.

This configuration is essential for maintaining code quality and preventing issues when files are transferred between different operating systems.

## .gitignore File Configuration
> File: `.gitignore`

The `.gitignore` file is configured to exclude specific files and directories from being tracked by Git. This file ensures that unnecessary and sensitive files, including environment configurations, Python build artifacts, IDE settings, system files, and test coverage data, are not included in version control.

### Ignored Files:
- **Virtual Environment**: Excludes `.venv/` to prevent committing the virtual environment directory.
- **Python Artifacts**: Excludes bytecode and distribution files like `__pycache__/`, `*.py[cod]`, `*.egg-info/`, `*.egg`, `dist/`, `build/`, and `*.whl`.
- **Environment**: Surrounding environment files such as `.env` and `.env.*` are ignored for security reasons.
- **IDE Configurations**: IDE-specific directories and temporary files like `.vscode/`, `.idea/`, and various swap files are ignored.
- **Operating System Files**: Common OS-specific files such as `.DS_Store` and `Thumbs.db` are excluded.
- **Testing and Coverage**: Excludes directories and files generated during test runs, like `.pytest_cache/`, `.coverage`, `htmlcov/`, and `.mypy_cache/`.
- **Project Planning**: The `planning.md` file is also ignored, likely for internal documentation that shouldn't be versioned.

This configuration facilitates a cleaner repository by avoiding the inclusion of temporary and system-specific files, thereby streamlining collaboration and deployment.

### Usage Examples

```bash
# Use Google Gemini
export GEMINI_API_KEY=your-key
codilay . -p gemini

# Use local Ollama (no API key needed)
codilay . -p ollama

# Use Groq with a specific model
codilay . -p groq -m mixtral-8x7b-32768

# Use a custom OpenAI-compatible endpoint
export CUSTOM_LLM_API_KEY=your-key
codilay . -p custom --base-url https://your-endpoint.com/v1 -m your-model

# Override any provider's base URL
codilay . -p openai --base-url https://your-proxy.com/v1
```

## CodiLay Dependency Graph
> File: `src/codilay/dependency_graph.py`

### CodiLay Dependency Graph

The `DependencyGraph` class is a core component of CodiLay, responsible for building a file-level Directed Acyclic Graph (DAG) from static import analysis across various programming languages. This feature allows CodiLay to determine which files can be processed in parallel and which must be executed sequentially based on their dependencies.

#### Key Features
- **Dependency Tracking**: The graph models dependencies using two dictionaries:
  - `depends_on[A] = {B, C}` where A imports from B and C
  - `depended_by[A] = {D}` where D imports from A

- **Processing Tiers**: Using topological sorting (Kahn's algorithm), it computes processing tiers. Files in the same tier can be executed in parallel.

- **Dependency Clusters**: The graph can identify clusters of files that do not depend on each other and can thus be handled independently.

- **Parallel Groups within Tiers**: Even within a single processing tier, files are further divided into groups if they share no direct dependency edges, allowing fine-grained parallel execution.

- **Import Extraction**: Supports multiple programming languages including Python, JavaScript, Go, Rust, Java, C/C++, Ruby, PHP, and Elixir. Employs regular expressions to extract import paths from source files and resolves these paths against known files in the project.

- **Statistics and Diagnostic**: Offers methods like `get_stats` to provide insights on the dependency graph, including edge counts, number of tiers, and cluster sizes.

The `DependencyGraph` class does not rely on any LLM calls which ensures its functionality remains fast and reliable, focusing solely on static analysis.

## CodiLay Search Module
> File: `src/codilay/search.py`

### CodiLay Search Module

The `search.py` module implements a single-file, full-text search engine designed for efficiently querying past conversation histories. This module does not require any external dependencies and employs a custom-built inverted index to facilitate rapid search operations.

#### Core Functionality

- **SearchResult and SearchResults Data Classes**: These classes structure individual and aggregate search results, including metadata like conversation titles, message roles, and relevance scores.

- **Stop Words & Tokenization**: Utilizes `_STOP_WORDS`, a predefined set of common, non-informative words that are filtered out during text tokenization by the `_tokenize` function.

- **ConversationSearch Class**: A multi-function class providing capabilities to build an inverted index from conversation data and perform keyword-based searching. It uses TF-IDF (Term Frequency–Inverse Document Frequency) scoring for ranking search results based on relevance.
  
  - **build_index Method**: Scans conversation files to create an in-memory inverted index, caching document metadata and lengths.
  - **_index_message Method**: Processes each message, updating the inverted index with token frequencies and computing normalized term frequency.
  - **search Method**: Executes a search over all indexed messages, returning the top-k results based on TF-IDF scoring. Supports filtering by message role and conversation ID.
  - **_make_snippet Method**: Extracts a portion of message content surrounding matching query terms, providing context as a snippet in search results.

#### Efficiency Considerations

- **TF-IDF and Normalization**: Scoring involves calculating TF-IDF for each query term, normalizing scores by document length to ensure fair comparison across messages of varying lengths.
- **Persistent Index Storage**: The inverted index can be saved and loaded from disk to avoid re-indexing, saving computational resources for subsequent searches.

This module is integral to facilitating fast, full-text search capabilities in CodiLay, enhancing user experience by enabling efficient retrieval of relevant past conversation snippets.

## CodiLay Triage System
> File: `src/codilay/triage.py`

### CodiLay Triage System

The `triage.py` module in CodiLay is responsible for the pre-flight classification of files within the system, an integral step in determining which files are most critical for processing and documentation. It supports two operational modes: `fast` and `smart`.

- **Fast Mode**: This mode uses static patterns to classify files instantly without leveraging AI models. It is designed for speed and zero cost. The classification relies on predefined sets of filenames, directory names, and file extensions to decide whether files should be classified as core, skim, or skip.

- **Smart Mode**: In this mode, the file tree (not the content) is passed to an AI model. The AI dynamically determines the importance of files, making a more nuanced assessment that can greatly enhance efficiency by focusing efforts where they matter most. This mode defaults to fast if the AI cannot be used or throws an error.

**File Categories**:

- **Core**: Files that are central to the application's operation and must be fully processed.
- **Skim**: Files that need a superficial review or limited processing.
- **Skip**: Files that are not necessary for processing, often because they contain build artifacts, libraries, or configuration.

**Static Patterns**: The module uses various hardcoded patterns to determine classification in fast mode:

- **ALWAYS_SKIP_DIRS**: A set of directory names such as `.git`, `node_modules`, etc., that should always be skipped.
- **ALWAYS_SKIP_EXTENSIONS**: A set of file extensions for files such as `.min.js`, `.pyc`, etc., that indicate build artifacts or minified files, which should be skipped.
- **ALWAYS_SKIM_FILES**: Specific configuration files like `package.json`, `setup.py`, etc., which are skimmed as they are important but don't require detailed analysis.

The `Triage` class manages the classification logic and applies a safety net, ensuring that files known to be nonessential are never incorrectly classified as critical. Furthermore, the `_detect_project_type` method is used to quickly identify the software stack based on the presence of specific files, aiding in a more directed triage process.

The module also includes utilities like `estimate_tokens_saved` for calculating resource savings by excluding particular files from processing based on average token counts.

Key functions:
- `fast_triage(all_files)`: Executes the fast triage logic.
- `smart_triage(file_tree, all_files, md_contents)`: Executes the AI-supported triage logic.
- `move_to_core(paths)`, `move_to_skip(paths)`, `move_to_skim(paths)`: Functions for reclassifying files between categories.

→ See [Triage Feedback System](#triage-feedback-system) for mechanisms to override and refine triage decisions based on user feedback.

## Parallel Orchestrator Module
> File: `src/codilay/parallel_orchestrator.py`

The Parallel Orchestrator module implements a tier-based parallel processing engine for file documentation. It ensures orderly and efficient handling of files according to their dependencies and allows both parallel and sequential execution based on tiers derived from the dependency graph.

### Architecture
The orchestrator's architecture is centered around a `Central Wire Bus` which synchronizes multiple `Worker` threads. Each worker updates a `DocStore` with locked access to maintain consistency and prevent race conditions.

1. **Central Wire Bus**: Manages and synchronizes wire state changes in a shared, thread-safe manner.
2. **Workers**: Run as threads, each assigned a file. They obtain a 'frozen' snapshot of the wire state to work independently yet safely.
3. **DocStore**: Responsible for storing documentation results from workers and reconciling them across tier boundaries.

### Processing Flow
1. Build a dependency graph to identify processing tiers through import analysis.
2. Compute tiers using topological sorting for execution order.
3. For each tier, process files:
   - Files within the same tier are processed in parallel using a thread pool.
   - Workers process files, updating results back to the central DocStore.
   - At tier boundaries, contexts are synchronized across all wires.
4. Process parked files sequentially once full context is available.
5. Finally, run a sequential self-review across generated sections to ensure consistency and completeness.

### Key Guarantees
- Parallel execution only occurs for files without interdependencies.
- Frozen wire snapshots ensure stability and consistency during processing.
- Wire mutations are safely handled through the locked `WireBus` interface.
- Tier boundary synchronizations maintain coherent document and wire states.
- Sequential passes review outputs from parallel sessions to catch inconsistencies.
